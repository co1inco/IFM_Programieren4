

# Debatte Viel / Wenig Dokumentation
## Für wenig Dokumentation sprechen:

 * Weniger arbeitsaufwand 
 * Duplizierter "Code". Die Doku ist evtl. ein Duplikat einer, sich bereits selbst erklärenden Funktion. Eine Änderung würde doppelte Arbeit bedeuten.
 * Keine falsche Dokumentation. 

## Für viel Dokumentation:

 * Weniger Fehler / Probleme Frühzeitig erkennen. Wenn man etwas Designed macht man sich meist mehr Gedanken um das "drumherum", zb. wie meine Komponente zukünftig mit anderen Komponenten interagieren sollen. Wenn man Implementiert, tendiert man dazu, nur die aktuelle Komponente zu betrachten. Daher kann es später zu Problemen und ggf mehr Arbeit kommen.
 * Zum Teil wird eine umfangreiche Dokumentation vorausgesetzt. zb. in Sicherheitsrelevanten bereichen
 * Durch eine ausgiebige Dokumentation wird das Onboarding neuer Personen vereinfacht. Ohne müsste eine neue Person ggf. "gebaybisittet" werden.
 * KI. Kann die Doku einerseits einfach und schnell Generieren. Außerdem kann sie besser Code generieren, wenn der bestehende Code gut Dokumentiert ist.
Wen eine ausgiebige Dokumentation von externe Seite verlangt wird, kann man da gegen nur wenig machen. (Der TÜF gibt sein ok nicht wodurch das Produkt nicht verkauft werden kann)

## Ansonsten kann man einen mittelweg finden.

 * Muss ein Meeting dokumentiert werden, oder recht es, dem fehlende Kollegen später davon zu berichten.
 * Wie lesbar ist der Code. Wie viel mehraufwand ist es für jemanden sich einzuarbeiten.
 * Kann ein Online-Meeting aufgenommen werden. Kein aufwand
 * Wer benötigt die Dokumentation. Geht es um ein öffentliches Projekt (Open source, öffentliche library, ...) oder ist es ein internes Projekt, mit wenig Onboarding. Bei letzterem ist warscheineich keine große Dokumentation notwendig und kann persönlich von den Kollegen gemacht werden


 # Scrum-Rollen-Spiel

 * Grundsätzlich haben sich die Rollen etwa so angefühlt, wie sie es sollten. Ich habe als PO zuerst die 3 stories vorgestellt und sie etwas im Detail ausgeführt.
 * Ich denke die Priorisierung war in Ordnung. Ich habe Nachrichten senden und empfangen mit einer gleichwertigen Pio versehen. Der Zeitstempel hatte eine geringere Priorität. Dies habe ich getan, da ich von außen eigentlich gar keine richtige Priorität festlegen kann (Das eine bringt nichts ohne das andere). Stadtessen hat das Entwicklerteam die Entscheidung getroffen, welches zuerst gemacht wird.

 * Die Aufteilung in stories war Ok. Das Entwicklerteam hat fragen bezüglich der Implementierung  gestellt (Farbe, Font, Ausrichtung). Bei der Aufteilung hat sich allerdings herausgestellt, dass zuallererst ein Projekt aufgesetzt werden muss. (Repository erstellen, Projekte erstellen, Templates erstellen). Da man diesen Schritt nicht klar auf eine der existierenden Stories legen kann, kam die Rückmeldung, dass hier evtl. eine weitere Story sinnvoll wäre, die dan die höchste Priorität hat. Danach wurden die Stories in einzelne Aufgaben unterteilt: Api implementieren, UI designen, UI Logik entwickeln, ...

* Der Scrum-Master war eher zurückhalten. Er hat zwar zwischendurch eingegriffen, wenn eine Diskussion ausgeartet ist, war aber ansonsten überfordert mit der Rolle
